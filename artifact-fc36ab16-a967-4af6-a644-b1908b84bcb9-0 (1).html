Aufgabenstellung – ITD Lösungsportal

Aufgabenstellung

Einfaches internes Portal für ITD-Lösungen (HTML/CSS/JS)

Ziel

Erstelle eine einfache, statische Website, die als internes Portal für ITD-Lösungen dient. Das Portal soll alle Lösungen (Agents, Tools, Prototypen etc.) übersichtlich anzeigen und grundlegende Informationen zu jeder Lösung bereitstellen. Die Website dient ausschließlich zu Übungszwecken, muss aber funktional und konsistent sein.

Technischer Rahmen

  • Entwicklungsumgebung: VS Code Web (https://vscode.dev/)
  • Technologien: Reines HTML, CSS und JavaScript (kein Framework, kein Build-Tool)
  • Keine Backend-Logik, kein Server, keine Datenbank
  • Die Seite muss lokal im Browser über index.html ausführbar sein.

Verzeichnisstruktur (fix vorgegeben)

  • index.html – Startseite / Portal
  • style.css – zentrales Stylesheet
  • app.js – JavaScript für Interaktivität
  • data/solutions.json – Beispiel-Datenbestand der Lösungen
  • img/ – optionaler Ordner für Logos/Icons

1. Inhalte und Funktionen

1.1 Startseite (index.html)

Erstelle eine Seite mit folgendem Aufbau.

Pflichtbestandteile

  1. Kopfbereich (Header)
    • Titel des Portals: ITD Lösungsportal
    • Untertitel: Zentrales Portal für interne Agents, Tools und Prototypen
  2. Navigationsbereich
    • Einfache Navigation mit mindestens:
      • „Übersicht“ (führt zur Startseite / Listenansicht)
      • „Über das Portal“ (Anker/Section auf derselben Seite ist ausreichend)
    • Navigation muss sichtbar und konsistent sein.
  3. Filter-/Suchbereich
    • Text-Suchfeld, mit dem die Liste der Lösungen live gefiltert wird (mindestens nach Name und Beschreibung).
    • Dropdown zur Filterung nach Status mit mindestens:
      • „alle“
      • „produktiv“
      • „prototyp“
      • „in Entwicklung“
    • Die Filterung muss ohne Neuladen der Seite funktionieren (JavaScript).
  4. Listenansicht der Lösungen
    • Die Lösungen werden in einer Karten- oder Tabellenansicht angezeigt.
    • Jede Lösung zeigt mindestens folgende Informationen:
      • Name der Lösung
      • Kurzbeschreibung (1–3 Sätze)
      • Ansprechpartner (Name oder Team)
      • Status (z. B. „produktiv“, „prototyp“, „in Entwicklung“)
      • Kategorie (z. B. „Agent“, „Tool“, „Prototyp“)
      • Link (z. B. zu Dokumentation oder Anwendung; kann ein Platzhalter-Link sein, z. B. https://example.com)
    • Es müssen mindestens 8 Beispiel-Lösungen angezeigt werden (über data/solutions.json).
  5. Detailansicht (optional, aber empfohlen)
    • Beim Klick auf eine Lösung (z. B. auf den Namen) wird eine Detailansicht angezeigt.
    • Varianten:
      • Anzeige in einem Overlay/Modal, oder
      • Anzeige im unteren Bereich der Seite.
    • In der Detailansicht müssen zusätzlich zur Kurzansicht angezeigt werden:
      • ausführlichere Beschreibung (mindestens 3–5 Sätze)
      • ggf. weitere Infos (z. B. verwendete Technologien, zuständiger Bereich)
  6. Bereich „Über das Portal“
    • Kurze Beschreibung des Zwecks:
      • zentrales Portal für ITD-interne Lösungen
      • Transparenz über vorhandene Lösungen
      • Förderung von Wiederverwendung
    • Klarstellung: „Dieses Portal ist ein leichtgewichtiger Einstiegspunkt und ersetzt keine CMDB.“

1.2 Datenhaltung (data/solutions.json)

Erstelle eine Datei data/solutions.json mit folgendem Aufbau:

[
  {
    "id": "solution-1",
    "name": "Beispiel-Agent 1",
    "descriptionShort": "Kurze Beschreibung der Lösung.",
    "descriptionLong": "Längere Beschreibung mit 3–5 Sätzen.",
    "owner": "Team XYZ",
    "status": "produktiv",
    "category": "Agent",
    "link": "https://example.com/agent1"
  }
  // … mindestens 8 Einträge insgesamt
]

Vorgaben:

  • Verwende die Felder exakt in dieser Schreibweise: id, name, descriptionShort, descriptionLong, owner, status, category, link.
  • Status-Werte müssen aus einem festen Set kommen (z. B. produktiv, prototyp, in Entwicklung).
  • Die Daten werden in app.js per fetch aus dieser JSON-Datei geladen.

2. Funktionale Anforderungen (JavaScript in app.js)

Pflicht

  1. Laden der Daten
    • Beim Laden der Seite werden die Einträge aus data/solutions.json geladen.
    • Die Lösungen werden anschließend automatisch in der Listenansicht angezeigt.
  2. Suchfunktion
    • Textfeld für Suche.
    • Während der Eingabe werden die angezeigten Lösungen live gefiltert.
    • Es muss mindestens in name und descriptionShort gesucht werden.
  3. Status-Filter
    • Dropdown mit mindestens folgenden Werten:
      • alle
      • produktiv
      • prototyp
      • in Entwicklung
    • Auswahl eines Status zeigt nur passende Lösungen an.
    • Die Status-Filterung muss mit der Textsuche kombinierbar sein (UND-Verknüpfung).
  4. Darstellung
    • Die Liste wird dynamisch im DOM erzeugt (kein Hardcoding der Lösungen in HTML).
    • Für jeden Eintrag wird eine eigene Karte oder Tabellenzeile erzeugt, basierend auf den JSON-Daten.

Optional (empfohlen)

  • Detailansicht beim Klick auf eine Lösung (siehe oben).
  • Anzeige, wie viele Lösungen aktuell im Filter-Ergebnis sind (z. B. „X Lösungen gefunden“).

3. Layout- und Designanforderungen (CSS in style.css)

Pflicht

  1. Grundlayout
    • Klare Struktur:
      • Header
      • Navigationsbereich
      • Filterbereich
      • Inhaltsbereich (Listenansicht)
    • Die Seite muss auf gängigen Desktop-Auflösungen gut lesbar sein.
  2. Responsivität (mindestens Basis)
    • Bei schmalen Bildschirmen (z. B. < 768 px) sollen Karten/Tabelle sinnvoll umbrechen.
    • Navigation muss weiterhin nutzbar bleiben (auch einfaches Untereinander-Anordnen reicht).
  3. Gestaltung
    • Einheitliches Farbschema (maximal 3–4 Hauptfarben).
    • Klare visuelle Abgrenzung von:
      • Header
      • Filterbereich
      • Liste
    • Lesbare Schrift (Systemschrift oder gängige Webfonts).
  4. Markierung von Status
    • Status (z. B. produktiv, prototyp) soll optisch hervorgehoben werden, z. B.:
      • farbiger Badge / Label,
      • unterschiedliche Hintergrundfarben,
      • Rahmen oder Icons.

4. Nicht erforderlich / ausdrücklich ausgeschlossen

  • Keine Benutzeranmeldung, kein Rollenmodell, kein echtes SSO.
  • Keine komplexen Freigabe- oder Genehmigungsworkflows.
  • Kein Backend, keine Datenbank, kein Servercode.
  • Keine Frameworks (kein React, Angular, Vue etc.).
  • Keine externen CSS-Libraries (kein Bootstrap, kein Tailwind etc.).

5. Abgabekriterien

Die Aufgabe gilt als erfüllt, wenn:

  1. Die Struktur wie vorgegeben existiert:
    • index.html
    • style.css
    • app.js
    • data/solutions.json
  2. Beim Öffnen von index.html im Browser:
    • Alle Lösungen aus der JSON-Datei geladen und angezeigt werden.
    • Suche und Status-Filter wie beschrieben funktionieren.
  3. Jede Lösung mindestens die geforderten Basisinformationen zeigt.
  4. Das Layout übersichtlich, konsistent und auf Desktop gut nutzbar ist.
  5. Der Code in HTML, CSS und JavaScript:
    • lesbar und übersichtlich ist,
    • mit sinnvollen Kurzkommentaren versehen ist (app.js),
    • und die Struktur in index.html grob kommentiert ist.
